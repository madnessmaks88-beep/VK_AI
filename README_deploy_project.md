# Math Tutor: от локальной LoRA-модели до продакшн VK-бота

Путь одного API от простого FastAPI-сервиса с локальной LoRA-моделью
до полноценного мультипровайдерного AI-ассистента, работающего в
ВКонтакте с памятью диалога, обработкой изображений и переключением
между несколькими LLM-провайдерами.

## Эволюция проекта — три итерации

### Итерация 1: локальная LoRA-модель (Qwen2.5-1.5B + LoRA-адаптер)
- FastAPI-сервис, загружающий дообученную модель из проекта LoRA
- Работает полностью локально, требует GPU
- **Проблема:** видеокарта (RTX 2060, 6 ГБ VRAM) сильно ограничивала
  размер и скорость модели

### Итерация 2: переход на Groq (облачный инференс)
- Полностью убрали локальную загрузку модели - сервис стал лёгким,
  не требует GPU вообще
- Подключили Llama 3.3 70B через бесплатный API Groq - на порядок
  быстрее и качественнее, чем то, что помещалось на локальную видеокарту
- **Компромисс:** больше не используется собственный LoRA-адаптер по
  математике - используется базовая модель общего назначения

### Итерация 3: мультипровайдерность + продакшн-фичи
- Добавлена поддержка DeepSeek (V4 Flash/Pro) как альтернативного
  провайдера - переключение между Groq и DeepSeek на лету через команду
- Понимание изображений (vision-модель) - можно прислать фото задачи
- Управление режимом "размышления" (thinking mode) у DeepSeek

## Финальная архитектура

```
ВКонтакте (пользователь)
        │
        ▼
  vk_bot.py (Long Poll API)
   ├── SQLite - память диалога (переживает перезапуски)
   ├── Rate limiting - защита от злоупотребления
   ├── Индикатор "печатает..."
   ├── Клавиатура с кнопками
   └── Обработка фото-вложений
        │
        ▼ HTTP (localhost:8000/ask)
  app.py (FastAPI)
   ├── Groq API  ──── llama-3.1-8b-instant ("fast")
   │              ──── llama-3.3-70b-versatile ("smart", по умолчанию)
   │              ──── llama-4-scout (vision, для фото)
   └── DeepSeek API ── deepseek-v4-flash
                    ── deepseek-v4-pro (+ управление thinking mode)
```

## Команды бота

| Команда | Действие |
|---|---|
| `/help` | Список всех команд |
| `/clear` | Очистить историю переписки |
| `/start` | Приветствие + сброс истории |
| `/model fast` / `/model smart` | Переключение между моделями Groq |
| `/model deepseek` / `/model deepseek-pro` | Переключение на DeepSeek |
| `/thinking on` / `/thinking off` | Режим размышления DeepSeek (вкл/выкл) |

Плюс: отправка фото с вопросом - автоматически обрабатывается через vision-модель.

## Инженерные решения и почему они приняты именно так

**SQLite вместо PostgreSQL** - для одного бота с умеренной нагрузкой
не нужен отдельный сервер БД, простой файл справляется полностью.

**Rate limiting на уровне пользователя, а не глобально** - защищает
от одного злоупотребляющего пользователя, не наказывая остальных.

**Разделение FastAPI-сервиса и бота на два процесса** - модель/провайдер
можно менять, не трогая логику интеграции с ВК, и наоборот. Тот же
`app.py` мог бы обслуживать и Telegram-бота, и веб-интерфейс одновременно.

**extra_body только для DeepSeek** - у Groq и DeepSeek разные SDK
(groq и openai соответственно) с разным набором поддерживаемых
параметров - код собирает аргументы вызова динамически, чтобы не
сломать один провайдер параметром, специфичным для другого.

**Явный проброс HTTP-статусов (429) от провайдера к клиенту** - вместо
того чтобы прятать все ошибки за общим 500, реальный статус превышения
лимита долетает до бота, который может показать пользователю осмысленное
сообщение и даже автоматически повторить запрос.

## Что публично доступно

- Веб-интерфейс с собственным дизайном ("тетрадь в клетку"), с плавными
  анимациями появления сообщений
- Публичный URL через ngrok (dev domain, не меняется между запусками)
- VK-бот в реальном сообществе

## Известные ограничения (честно, для дальнейшего развития)

- История диалога в ВК хранится в SQLite локально - если задеплоить
  бота на другой сервер, файл базы нужно перенести/пересоздать
- Rate limiting и выбор модели пользователем хранятся только в
  оперативной памяти процесса бота - обнуляются при перезапуске
  (в отличие от истории диалога, которая в SQLite)
- Нет автоматического перезапуска при падении процесса - в продакшене
  стоило бы обернуть в systemd/nssm или использовать supervisor
- Бот работает только в личных сообщениях, не в групповых беседах

## Как запустить

```bash
pip install fastapi uvicorn groq openai vk_api requests

# Терминал 1
set GROQ_API_KEY=твой_ключ
set DEEPSEEK_API_KEY=твой_ключ
python app.py

# Терминал 2
python vk_bot.py
```

## Структура проекта

```
├── app.py              # FastAPI-сервис, мультипровайдерная генерация
├── vk_bot.py            # VK-бот: Long Poll, команды, память, rate limiting
├── db.py                # SQLite-хранилище истории диалогов
├── static/index.html    # Веб-интерфейс чата
├── check_gpu.py         # Утилита проверки видеопамяти (осталась с итерации 1)

```
